## Course bridge

TLS 1.3 begins negotiation with ClientHello and ServerHello; the client offers
capabilities while the server selects acceptable parameters.[^tls]

[^tls]: RFC 8446 §§4.1 and 4.1.2.

Random, session, ordering, extension, cipher, group, signature, version, and ALPN
behavior creates observable structure. An intermediary terminating TLS produces
its own downstream handshake, so observation point matters.

## Worked example

The helper uses Python `ssl.MemoryBIO`: OpenSSL produces the first handshake bytes
without a network socket. Adding ALPN changes the generated byte sequence, but
random/session material may also differ, so the raw digest difference is not a
controlled JA4 comparison.

## Guided exercise

### Objective

Generate and annotate two local ClientHello fixtures.

### Setup

Read `lab/protocol/compare.py`; it parses only outer record and handshake fields.

### Actions

1. Execute `python -m lab.protocol.compare clienthello`.
2. Verify record type 22 and handshake type 1.
3. Explain record versus ClientHello legacy versions.
4. Compare lengths/digests and record ALPN as the declared configuration change.
5. List uncontrolled per-handshake fields and prohibited conclusions.

### Expected output

Two ClientHello summaries with different bytes/digests and explicit non-JA4,
non-identity limitations.

### Interpretation

Configuration affects protocol shape. A robust fingerprint method normalizes or
selects fields deliberately; this helper does not claim to implement one.

### Common failure modes

- Calling the SHA prefix JA4
- Treating legacy version field as negotiated version
- Ignoring random/session variation
- Inferring browser identity from Python/OpenSSL bytes

### Cleanup

No socket opened and no service persists.

## Why this matters offensively

A browser environment claim can conflict with the TLS stack seen at the edge.
Understanding the handshake reveals which layer a proxy or alternate client changes.
